## Mission

Compile LĂ–VE's actual C++ engine â€” real bindings, real Box2D, real decoders, real font stack â€” to `wasm32-wasi`, so that a browser can run a LĂ–VE project with engine behavior that *is* desktop LĂ–VE's, not an imitation of it. The artifact must:

- **Run real LĂ–VE source.** Not a JS reimplementation of the `love.*` API. A reimplementation was considered and rejected: multi-year effort, and never bit-exact. The fidelity bar is concrete â€” e.g. every Lua-facing engine call funnels through `luax_catchexcept`'s typed C++ exception handling (145 call sites); imitations get details like this subtly wrong forever.
- **Need no Emscripten, no pthreads, no SharedArrayBuffer, no COOP/COEP headers.** The IDE must be able to run its preview on any static host. (love.js requires cross-origin isolation because its engine build bakes in `-pthread`; that is unremovable by swapping the Lua VM â€” it's why this project exists.)
- **Keep the project tree `.love`-compatible.** The same game source runs unmodified on desktop LĂ–VE â€” a game made in the browser can go to desktop and back. The **primary target is a LĂ–VE game that runs natively in the browser**; a high-fidelity **desktop preview** is the second, valued use of the same engine. (The fused per-game web artifact for standalone *shipping/distribution* is a later packaging step, not yet built â€” but browser-native games are a **first-class target**, no longer a deferred descope.)
- **Prefer faithful primitives over emulation** where the browser has the real thing (real Web Workers for `love.thread`, not coroutines pretending to be threads).

## The fidelity standard â€” browser-native correctness first

Two use-cases share this one engine, and they set the priority order every seam decision is measured against:

1. **A correct browser game â€” the must-hit bar.** As a *browser* game, the engine must be complete and correct; this is achievable and held to 100%.
2. **`.love` source-compatibility â€” a pillar.** The same source runs unmodified on desktop LĂ–VE.
3. **Desktop *behavioral* parity â€” aspirational, the reference, not the pass/fail line.** The browser genuinely cannot match desktop 100% (async-storage durability, HRTF, mic rates, threading), and nobody expects it to; every gap is **declared, never faked**.

So each decision is judged by *"what does a correct browser game do?"*, not *"does it byte-match desktop?"* â€” desktop is the reference, browser-native correctness is the standard. This generalizes the audio seam's principle (`wasi/audio/DESIGN.md`, Decision 3: *"the bar is device-agnostic fidelity, not desktop parity"*) to the whole engine.

## What stays real, what gets touched â€” the honest claim

The semantically hard code stays verbatim: physics, decoders, render math, module logic, the Lua bindings. The platform-adjacent plumbing gets touched: backend selection, internal thread usage (audio pump, timers) massaged into a single-threaded frame-pump model, and the build system. Expect the diff against upstream to be the evidence â€” small, seam-shaped, and reviewable â€” rather than a "95% unmodified" slogan.

## Toolchain

- `clang-20+` + `wasi-libc`, C++ with **`-fwasm-exceptions`** and the *standardized* wasm-EH encoding â€” matching lua-wasi's toolchain, which is mandatory: the VM and this engine share one EH machinery, so the LLVM major and EH encoding must agree. Caution, probed 2026-07-07: clang-20's bare `-fwasm-exceptions` **defaults to the legacy encoding**; the standardized one needs an explicit `-mllvm -wasm-use-legacy-eh=false`, which is baked into `wasi/toolchain/build-libcxx-eh.sh` and every build script here â€” and enforced per artifact by `wasi/toolchain/check-eh-encoding.sh` (disassembly must show `try_table` and zero legacy forms; engines accept both encodings, so only a build-time gate can catch a lost flag). LĂ–VE's own error path requires full C++ EH â€” typed catches and exception-object destructors â€” so the build vendors **LLVM libc++ + libc++abi compiled with wasm-EH** (wasi-sdk's stock libc++ is built without exception support). Wasm `setjmp`/`longjmp` â€” which Ubuntu's `wasi-libc` omits entirely but FreeType needs â€” is vendored into the same sysroot (`wasi/toolchain/setjmp`, from wasi-libc); on wasm it is implemented *on top of* wasm-EH, so it rides on the one encoding and needs only the per-TU flag `-mllvm -wasm-enable-sjlj` (single-sourced as `$SJLJ_FLAGS`).
- **Lua VM:** [Lua2D/lua-wasi](https://github.com/Lua2D/lua-wasi) (Lua 5.4 + selective AOT), consumed as a **source drop at a pinned commit**, compiled in-tree with this build's own flags, with `LUAW_EXTERNAL_EH` so the real libc++abi owns exception dispatch. LĂ–VE 12 supports Lua 5.4 natively (`LUA_VERSION_NUM >= 504` paths in `love.cpp`, `common/runtime.cpp`); LuaJIT is not an option under wasm (no runtime codegen; no wasm interpreter backend).
- **No Emscripten anywhere.** The browser side is a small hand-written WASI preview1 shim (`fd_write`, clocks, `proc_exit` â€” a few dozen lines) plus the import surface defined by the seams below.

## The three seams (new code)

LĂ–VE has always delegated exactly these to the host OS; a browser tab is just a different host:

1. **Graphics/window.** Upstream: SDL3 provides window/context; `graphics/opengl` issues GL calls. Here: no SDL, no EGL. Because WebGL2 *is* OpenGL ES 3.0 â€” which LĂ–VE's `opengl` backend already targets â€” the backend is **reused, not rewritten**: it is reseamed under one guarded `LOVE_GRAPHICS_GL_STATIC_IMPORTS` macro (byte-unchanged for default builds, offered upstream) across two files â€” the GL loader in `OpenGL.cpp` becomes **static WebGL2 imports** so the host's WebGL2 context is the GL surface (and, next to it, LA8 is reported unsupported so the font atlas falls back to RGBA8, since WebGL2 lacks the texture swizzle LĂ–VE's LA8 path uses), and `StreamBuffer.cpp` selects the `glBufferSubData` streaming strategy because WebGL2, unlike the GLES it reports itself as, forbids client-side vertex arrays and buffer mapping. **Sub-steps 4.1â€“4.20 landed:** `love.graphics` clears a real framebuffer (4.1), draws a positioned filled rectangle (4.2), draws the rest of the 2D primitives in one frame (4.3), draws a real texture built from CPU pixels (4.4), compiles and runs a **user** pixel shader (4.5), renders into an off-screen **canvas** then samples it back (4.6), prints **text** with the embedded default font (4.7), composites/clips with the cross-cutting **render state** â€” blend, scissor, stencil (4.8), draws the higher-level drawables â€” Mesh (4.9), SpriteBatch + Quad (4.10), ParticleSystem (4.11) â€” and then the compose-only API tail: coordinate transforms (4.12), multiple render targets (4.13), a multisampled canvas (4.14), the engine's own texture readback (4.15), and an explicit GraphicsBuffer drawn (4.16), plus the extended tail â€” instanced drawing (4.18), the depth buffer + depth test (4.19), and a non-default ImageFont built from an image, distinct from FreeType (4.20) â€” all through the real backend in Chromium: the full 2D drawing surface, its compositing state, the drawables, and the extended draw API. What's out of reach is the WebGL2 ceiling (compute shaders, indirect draw, SSBO â€” genuinely absent on ES 3.0), witnessed as a graceful, declared divergence (4.17, #36): the engine reports them unsupported and rejects a compute shader with a catchable error rather than crashing. `love.window` landed at step 6.3 â€” a host-import backend whose `setMode` creates the real `<canvas>` + WebGL2 context â€” which unblocked a real `present()` and with it the window-dependent `captureScreenshot`, completing step 4.
2. **Audio.** Upstream: OpenAL device output (plus a mixing/streaming thread). Here: per-source PCM pushed to **WebAudio** via imports â€” the host mixes and resamples, so love-wasi owns no DSP. **Landed** on the `webaudio` backend (playback + microphone, tone-recovery witnessed on node + Chromium); `audio/null` is the always-linked fallback. Decoding (vorbis, flac, mp3, wav) stays real, in-tree C â€” not yet compiled into this build.
3. **`love.thread`.** wasm32-wasi is single-threaded; real threading means real **Web Workers** with message-passing. LĂ–VE's Channel API is share-nothing by design, so semantics mostly survive â€” but this is an honest, documented behavioral divergence, not a bug to hide.

Everything else the host supplies as imports, which is the same role an OS plays for desktop LĂ–VE: `love.filesystem` backed by the IDE's project storage (replacing PhysFS), input events forwarded from the DOM into LĂ–VE's real event queue, and the frame pump driven by `requestAnimationFrame` (the engine runs as a resident coroutine; the host resumes it once per frame â€” this repo owns its own pump against lua-wasi's embedding surface; lua-wasi's `onelua.c` reactor glue is not used or extended).

## conf.lua

`love.conf(t)` is parsed and honored identically to desktop: `t.window.*` drives the canvas and page title; `t.modules.*` gates which subsystems must exist for a given game (so a game with `t.modules.physics = false` previews before the Box2D link lands); `t.identity` namespaces save data. Settings with no browser equivalent (`t.window.display`/`x`/`y`, exclusive fullscreen, multi-window) are explicitly mapped or no-op'd, and documented â€” never silently faked.

## Upstream relationship â€” branches, patches, and the 12.0 swap

LĂ–VE 12 is unreleased but functionally complete (its open milestone items are polish, several in backends this build excludes). This fork does not wait for it, does not finish it, and must be able to adopt the released 12.0 cheaply. The machinery:

**Branch model.** `main` is a pristine mirror of upstream `love2d/love` â€” never committed to, only fast-forwarded when adopting a new upstream commit. **`wasi` is the working branch and the repo default**; everything this fork adds lives there. At any moment, `git diff main...wasi` is the complete, current answer to "what did this fork touch" â€” the evidence for the honest claim above.

**Three lanes for changes:**

1. **Wasi-only code** â€” new backends, build files, the pump. Additive by design (new files, minimal edits to shared code), lives on `wasi` forever, rebases near-conflict-free.
2. **Generic fixes LĂ–VE needs** â€” cut a topic branch from the *upstream base commit* (not from `wasi`), PR it to love2d, cherry-pick it into `wasi` meanwhile. When upstream merges, the carried copy is shed automatically at the next rebase (git recognizes already-applied patches). Carried, never kept.
3. **Fork-private edits to shared engine code â€” not allowed.** If upstream declines a fix this port needs, that becomes a recorded design decision here, never a silent divergence.

**The swap.** Adopting a new upstream â€” a fresh nightly or the 12.0 release itself â€” is: fast-forward `main` to the new commit, rebase `wasi` onto it, watch lane-2 patches fall out, update the base pin at the top of this README. Because lane 1 is additive and lane 2 self-sheds, the expected cost is an afternoon, not a migration.

## Constitution

- **One artifact, no parts.** The shipped form is a single `.js` file with the wasm embedded; Workers spawn from Blob URLs. A repo may have a build; its output may not have pieces.
- **Pin-by-commit in both directions.** Upstream base pinned above; lua-wasi pinned in-tree; LoveIDE pins this repo's artifact by commit. Nothing floats.
- **Claims match evidence.** A subsystem "works" when the `testing/` corpus exercises it in a real browser and matches desktop behavior. Anything less is labeled as less.
- **Upstreamable by construction.** The web backend is written against upstream's own abstractions; small generic seams are offered upstream as they arise; the goal-state conversation â€” a web platform for LĂ–VE proper â€” stays open.
